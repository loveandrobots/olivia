# Olivia — Data Freshness & Gentle Review
## Visual Implementation Spec

*March 2026 · Design handoff for implementation*

---

> **Scope of this document**
>
> This spec defines every UI state required to implement the Phase 2 Data Freshness feature (approved D-058, 2026-03-17). It covers three new surfaces:
>
> 1. **Monthly household health check** — a home screen prompt card + a dedicated review screen with grouped stale items, per-item actions, completion celebration, and partial progress
> 2. **Freshness nudge cards** — a new `freshness` nudge type in the existing H5 nudge tray, visually distinguished as informational (not time-sensitive)
> 3. **"Still active?" confirmation pattern** — the single-tap confirmation and archive confirmation dialog used across both surfaces
>
> **Design philosophy:** These surfaces must feel **calm and helpful, not nagging**. The health check is a quick pulse (under 2 minutes), not an audit. "Still active" is always the easiest action. The spec follows the product ethos: "reduce cognitive load, not increase it."

---

## 0. Design Decisions

### 0.1 Health Check Card Placement on Home Screen

**Decision:** The health check card appears in the **Today section hero zone** of the Today-Forward home screen (D-055), positioned **below the nudge tray** (if present) and the existing Olivia AI voice card (if present), but **above the Today workflow sections**.

**Rationale:** The health check is a monthly housekeeping prompt — important but not urgent. It should not compete with time-sensitive nudge cards or the AI voice suggestion. Placing it between the proactive cards and the daily content gives it visibility on the first scroll without creating urgency. It lives in the same flow as "things Olivia noticed" rather than in the workflow content.

### 0.2 Health Check Card — Warm Informational, Not Nudge-Styled

**Decision:** The health check card uses a **sky (`--sky`) accent** — visually distinct from all existing nudge card types (mint for routines, rose for reminders, violet for planning rituals). It is a surface card, not a gradient AI voice card.

**Rationale:** The sky accent reads as informational and calm — it's the only accent color not already assigned to a workflow-type nudge. Using mint, rose, or violet would create confusion with the existing nudge card types. The health check is not a nudge (it's not generated by `GET /api/nudges`); it needs its own visual identity.

### 0.3 Health Check Review Screen — New Dedicated Screen

**Decision:** The health check review screen is a **full-screen overlay** accessible by tapping the home screen health check card. It renders as a scrollable list of stale items grouped by workflow type, with per-item action buttons. A back/close button returns to the home screen.

**Rationale:** The health check contains up to 10 items across multiple workflow types. This requires more space than an inline card or a modal. A dedicated screen respects the 4-tab navigation model by overlaying the Home screen rather than adding a new tab. The user's mental model: "I opened the health check from home, I close it and I'm back at home."

### 0.4 Freshness Nudge Visual Treatment — Informational Tier

**Decision:** Freshness nudge cards use a **sky (`--sky`) accent** with left stripe and icon container, matching the health check card's accent. Their card anatomy matches the existing nudge card pattern (§3 of the nudges visual spec) but with sky-tinted visual elements. This gives freshness nudges a visually distinct, lower-urgency feel compared to routine (mint), reminder (rose), and planning ritual (violet) nudges.

**Rationale:** The feature spec states: "Freshness nudges rank below all existing nudge types. Freshness is informational; the existing types are time-sensitive." A distinct accent color signals this lower urgency visually. Sky is calm and neutral — it reads as "FYI, when you have a moment" rather than "action needed now."

### 0.5 Archive Confirmation Dialog

**Decision:** Archive uses a **compact confirmation card** that appears inline where the item was, replacing the item row temporarily. Not a full-screen modal or a system dialog.

**Rationale:** Olivia has no modals in the core navigation model. An inline confirmation keeps the user in flow and avoids the weight of a system dialog for what should feel like a lightweight action. The confirmation is brief: item name + two buttons (Cancel / Archive).

### 0.6 "Still Active" — Single Tap, No Feedback Dialog

**Decision:** "Still active" is a single tap that immediately removes the item from the review list (or removes the freshness nudge card). A subtle checkmark animation provides confirmation. No toast, no dialog, no undo prompt.

**Rationale:** The spec is explicit: "Still active is always one tap — no confirmation dialog." The checkmark animation (200ms) provides visual confirmation that the tap registered without interrupting flow. Keeping this interaction frictionless is critical — users should tap through 10 items in under 2 minutes.

### 3.6 Action buttons

Each item row has three actions. Button layout is a horizontal row with even spacing:

```
[Still active]    [Archive]    [Edit →]
```

| Button | Style | Min height | Copy |
|---|---|---|---|
| **Still active** | `.btn-primary` — `background: var(--violet)`, `color: white`, `border-radius: 12px` | 40px | "Still active" |
| **Archive** | `.btn-secondary` — `background: var(--lavender-soft)`, `color: var(--violet)`, `border-radius: 12px` | 40px | "Archive" |

**"Still active" is the primary action** — leftmost, violet fill, largest visual weight. This is intentional per the spec: "Still active should be the easiest action."

**"Archive"** is secondary — lavender-soft fill, adequate but not dominant.

**"Edit →"** is tertiary — text-only link that deep-links to the item's edit screen. Minimal visual presence.

**Gap between buttons:** 8px minimum. All buttons should have 44px minimum tap target height (with padding).

## 4. Health Check Interactions

### 4.1 "Still active" tap (FRESH-CHECK-5)

Single tap. No confirmation dialog.

1. Button briefly shows a checkmark icon (✓) replacing the text, `background: var(--mint)`, 200ms
2. The item card collapses: `opacity: 1 → 0`, `height: current → 0`, `overflow: hidden`, duration 250ms, ease-out
3. If the workflow group is now empty, the section label also fades out (200ms)
4. Remaining cards slide up to fill the gap smoothly

**Reduced motion:** Item disappears immediately (no height animation). Checkmark still flashes briefly (100ms).

### 4.2 "Archive" tap — Inline confirmation (FRESH-CHECK-4)

Tapping "Archive" replaces the item card's action row with an inline confirmation:

```
┌─────────────────────────────────────────────┐
│  [icon]  Morning routine                    │
│          Last completed 3 weeks ago         │
│                                             │
│  Archive "Morning routine"?                 │  ← confirmation text
│  [Cancel]              [Archive]            │  ← confirmation buttons
└─────────────────────────────────────────────┘
```

| Element | Token(s) |
|---|---|
| Confirmation text | 13px PJS regular, `color: var(--ink)` |
| "Cancel" button | `.btn-secondary` — `background: var(--lavender-soft)`, `color: var(--violet)` |
| "Archive" button | `background: var(--rose)`, `color: white`, `border-radius: 12px` |

**Rose-colored archive button** signals the destructive nature of this action, consistent with the rose = urgent/attention visual language.

**Cancel:** Returns to the normal 3-button row. No data change.

**Confirm archive:** Item card collapses with the same animation as "still active" (§4.1). The entity receives the type-appropriate archival (resolved → paused → cancelled per spec).

### 4.3 "Edit →" tap

Navigates to the item's edit screen (deep-link). On return to the health check screen, the item is removed from the review list (considered addressed). Same collapse animation.

### 4.4 Completion state (FRESH-CHECK-3)

When all items have been reviewed (still-active'd, archived, or edited), the screen transitions to:

```
┌──────────────────────────────────────────────┐
│  ← Back           Monthly check-up           │
│                                              │
│                                              │
│              ✦                               │  ← Olivia orb (small, 48px)
│                                              │
│        All caught up!                        │  ← Fraunces 700 24px, --ink
│                                              │
│     Everything looks good. See you           │  ← Fraunces italic 15px, --ink-2
│     next month for another check.            │
│                                              │
│         [Back to Home]                       │  ← btn-primary, centered
│                                              │
│                                              │
└──────────────────────────────────────────────┘
```

**Celebration moment:**
- A small Olivia orb (48×48px) with `orbPulse` animation, centered
- "All caught up!" — Fraunces 700 24px, `color: var(--ink)`, centered
- Subtext — Fraunces italic 15px, `color: var(--ink-2)`, centered, line-height 1.5
- "Back to Home" button — standard `.btn-primary`, centered, 44px height

**Entry animation:** `fadeUp` (opacity 0→1, translateY 12→0), 300ms, ease-out. The orb, title, subtext, and button stagger at 80ms intervals.

**After returning to home:** The health check card is removed. `lastHealthCheckCompletedAt` is updated. The next card appears in ≥30 days.

### 5.5 Freshness Nudge Priority and Throttling

- Freshness nudges render **below** all existing nudge types in the tray (below routine, reminder, and planning ritual nudges)
- Maximum **2 freshness nudges per day** — separate from the existing nudge display cap
- If both freshness nudges and regular nudges are present, the regular nudges appear first in the tray, followed by freshness nudges
- Freshness nudges have no push notifications — in-app only

### 5.6 Freshness Nudge Copy Tone

Calm, questioning, non-judgmental. The copy always ends with a gentle question:

| Workflow type | Example copy |
|---|---|
| Routine | "Weekly cleaning hasn't been marked done in 3 weeks — still on track?" |
| Reminder | "Pick up dry cleaning was due 10 days ago — still need this?" |
| Shared list | "Grocery list hasn't been updated in 5 weeks — still using it?" |

**Copy rules:**
- Always include the item name
- Always include the staleness duration (human-readable relative time)
- Always end with a gentle question ("still on track?", "still need this?", "still using it?")
- Never accusatory: no "You haven't..." or "You missed..."
- Inbox items follow the existing stale nudge pattern (unchanged)
- Meal plans: no inline freshness nudges (health-check-only per D-058)

### 5.7 Freshness Nudge Actions

**"Still active" tap:** Same as health check — single tap, no confirmation. Updates `freshnessCheckedAt`, card animates out with standard nudge card exit (opacity fade + height collapse, 200ms).

**"Archive" tap (FRESH-NUDGE-4):** Shows inline confirmation within the card:

```
┌─────────────────────────────────────────┐
│ ┃  ┌────┐                              │
│ ┃  │ ↻  │  Weekly cleaning              │
│ ┃  └────┘                               │
│ ┃                                      │
│ ┃  Archive "Weekly cleaning"?           │
│ ┃  [Cancel]           [Archive]         │
└─────────────────────────────────────────┘
```

Confirmation button uses `background: var(--rose)`, `color: white` (same rose treatment as health check archive). On confirm, card exits. On cancel, card restores to normal state.

**Dismiss × tap:** Standard nudge dismiss — card hidden for rest of calendar day. Same exit animation as other nudge types.

## 7. State Transitions & Motion

### 7.1 Health check card entry (home screen)

- Card animates in with `fadeUp` (opacity 0→1, translateY 12→0), 250ms, ease-out
- Appears after nudge tray and AI voice card have loaded (stagger delay: 100ms after last nudge/AI card)

### 7.2 Health check screen entry

- Standard `screenIn` animation (opacity 0→1, translateY 10→0), 300ms
- Items stagger in with `taskIn` at 50ms intervals per card
- Section labels appear 30ms before their first card

### 7.3 Item review animations

- "Still active" checkmark flash: 200ms, then card collapse 250ms
- Archive inline confirmation: action row content crossfades (150ms)
- Archive confirmed: card collapse 250ms (same as still active)
- Section label removal (when group empty): 200ms fade-out

### 7.4 Completion celebration

- Remaining item list fades out (200ms)
- Orb, title, subtext, button stagger in with `fadeUp` at 80ms intervals
- Orb begins `orbPulse` animation immediately

### 7.5 Reduced motion

All animations degrade to instant show/hide. The orb pulse is disabled. Checkmark flash reduced to 100ms.

## 9. Accessibility Notes

- All action buttons maintain 44px minimum tap target height
- "Still active", "Archive", and "Edit →" have sufficient horizontal spacing (≥8px) to prevent accidental taps
- The health check review screen uses semantic HTML: `<section aria-label="Routines">` for each workflow group, `<button>` for all actions
- Color is never the sole differentiator — each workflow type uses icon + label + color together
- Archive confirmation uses explicit text ("Archive 'Morning routine'?") rather than relying on context alone
- Focus states use `var(--violet-glow)` which strengthens automatically in dark mode
- `prefers-reduced-motion` is respected for all animations (§7.5)
- The sky accent `#48CAE4` passes AA contrast requirements for both text and non-text elements on all surface tokens in both modes

## 10. Edge Cases

### 10.1 Zero stale items at health check time

If the monthly health check trigger fires but zero items are actually stale (all freshness clocks are current), skip the health check entirely. Do not show the card. Update `lastHealthCheckOfferedAt` and wait for the next cycle.

### 10.2 User opens health check, all items are already addressed

If items become stale between the card appearing and the user opening the review screen (e.g., the user completed routines in the meantime), show only the items that are currently stale at the time of screen open. If none remain, skip directly to the "All caught up!" celebration.

### 10.3 Only 1 stale item

The health check screen renders normally with a single item card. The single-item review should still feel complete — the intro text, the item, and the celebration after reviewing it.

### 10.4 Freshness nudge + health check card coexisting

Both can appear on the home screen simultaneously. The health check card sits below the nudge tray. Freshness nudges in the tray may reference some of the same items that appear in the health check. This is acceptable — the user can address them from either surface.

### 10.5 App restart during health check

Partial progress is saved in local Dexie state. On app restart, the health check card shows FRESH-HOME-2 (partial progress). Reopening goes directly to FRESH-CHECK-2 with already-reviewed items removed.

## 12. Open Questions

### For Founding Engineer

1. **Health check screen routing:** Should the review screen be a new route (e.g., `/health-check`) or a full-screen overlay component rendered within the Home route? A dedicated route is simpler for navigation state; an overlay preserves the home screen scroll position.

2. **Freshness nudge data source:** Are freshness nudges returned from the existing `GET /api/nudges` endpoint (extended), or a separate freshness-specific endpoint? The visual spec is agnostic to this — the card rendering is the same either way.

3. **Health check partial progress storage:** Local Dexie table recommended. The spec requires that progress survives app restart. The engineer should decide on the exact schema.

4. **Item cap adjustment:** The spec uses 10 items per health check. If real-world testing shows this is too many (review takes >2 minutes), the cap can be reduced without visual spec changes.
