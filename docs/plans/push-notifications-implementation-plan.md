# Push Notifications Implementation Plan

**Goal:** Implement H5 Phase 2 push notifications — extend the proactive nudge surface beyond in-app cards to OS-level device notifications. When a household member is not using Olivia, they receive a native push notification when an overdue routine, approaching reminder, or overdue planning ritual exists. Tapping the notification opens Olivia to the home screen (nudge tray). This is a household-wide, advisory-only delivery mechanism.

**Architecture:** The codebase already contains push infrastructure (`push.ts` — `WebPushProvider`/`DisabledPushProvider`, VAPID config, `jobs.ts` background scheduler) and a `notification_subscriptions` table for the inbox/reminder notification system. The H5 nudge push system is a **separate, coexisting** push layer: new `push_subscriptions` and `push_notification_log` tables, new `/api/push-subscriptions/*` endpoints, and a new nudge-specific scheduler rule — all built on the same `push.ts` provider and VAPID key config. The two systems share VAPID credentials but maintain independent subscription registries and dedup logs because they serve different use cases and have different data shapes (`userId` nullable column for Phase 3 targeting, `entityType`/`entityId` dedup semantics distinct from the reminder bucket system).

The service worker is currently generated by VitePWA with Workbox (`generateSW` strategy). Adding push event handling requires switching to `injectManifest` strategy, which generates precaching but delegates all other logic to a custom `sw.ts` file we control.

**Tech Stack:** TypeScript, Zod, Fastify, better-sqlite3, Drizzle ORM, React, TanStack Query, Dexie, Vitest, `web-push` (already installed).

**Spec dependency:** All product decisions are resolved in `docs/specs/push-notifications.md` (CEO-approved, D-045, 2026-03-15). Key D-045 resolutions:
- Scheduler interval: **30 minutes** (Founding Engineer may adjust)
- Dedup window: **2 hours**
- `push_subscriptions` includes a **nullable `userId` column** from the start
- iOS Home Screen note: **yes** in opt-in prompt
- Scheduler placement: **in-process recurring interval**
- VAPID contact email: placeholder acceptable, configurable via env var

## Architecture Decisions

### Decision A: Separate push_subscriptions table, not reusing notification_subscriptions

**Chosen approach:** Create a new `push_subscriptions` table for H5 nudge push subscriptions, separate from the existing `notification_subscriptions` table used by the inbox/reminder system.

**Rationale:** The two systems have different data shapes and lifecycle expectations:
- `notification_subscriptions` uses `actorRole` (stakeholder/spouse) as the scoping key and stores payload as a blob
- `push_subscriptions` scopes by `householdId` (household-wide per spec) and separates p256dh/auth into typed columns
- The nullable `userId` column is required from the start for Phase 3 per-member targeting — this can't be cleanly retrofitted onto `notification_subscriptions`
- Dedup semantics differ: nudge push deduplicates by `(entityType, entityId)`, reminder push deduplicates by delivery bucket

### Decision B: Shared VAPID credentials, new endpoint path

**Chosen approach:** The new `GET /api/push-subscriptions/vapid-public-key` endpoint returns `config.vapidPublicKey` — the same key used by the existing system. The endpoint path is `/api/push-subscriptions/vapid-public-key` as specified (distinct from the existing `/api/notifications/vapid-public-key` for the inbox system).

**Rationale:** Both systems use the same VAPID key pair (one key pair per server installation). Separate paths serve distinct PWA-side subscription flows and keep the H5 push feature self-contained.

### Decision C: injectManifest strategy for Service Worker

**Chosen approach:** Switch VitePWA from default `generateSW` to `injectManifest` strategy. Create `apps/pwa/src/sw.ts` that imports `precacheAndRoute` from Workbox and adds custom `push` and `notificationclick` event handlers.

**Rationale:** The existing `generateSW` strategy produces a Workbox-only service worker with no entry point for custom push event handling. `injectManifest` gives us a controlled `sw.ts` where Workbox injects the precache manifest at build time but we own all other SW logic. This is the recommended VitePWA pattern for custom event handlers.

**Implementation note:** Set `swSrc: 'src/sw.ts'` in the VitePWA config. The generated file goes to `dist/sw.js`. No change to `registerType: 'autoUpdate'`.

### Decision D: Nudge push scheduler as a new rule in jobs.ts

**Chosen approach:** Add `evaluateNudgePushRule` to `jobs.ts`. Wire it with its own 30-minute interval alongside the existing notification interval, or (preferred) as a second call block inside `startBackgroundJobs` with a separate `nudgePushIntervalMs` config value.

**Rationale:** The existing `startBackgroundJobs` runs at `config.notificationIntervalMs` (default 1 hour). The nudge push scheduler should run at a different interval (30 minutes). Rather than forcing all jobs onto the same tick, use a second `setInterval` inside `startBackgroundJobs` for nudge push.

**Implementation note:** Add `nudgePushIntervalMs?: number` to `AppConfig` (default: `1_800_000` — 30 minutes, from `OLIVIA_NUDGE_PUSH_INTERVAL_MS` env var). If VAPID keys are absent, `evaluateNudgePushRule` logs a warning and returns early without failing.

### Decision E: householdId hardcoded as 'household' in Phase 2

**Chosen approach:** Since Olivia is currently a single-household app with no multi-household concept, use `'household'` as the `householdId` value in `push_subscriptions` for Phase 2. The column exists in the schema and is enforced at the application layer.

**Rationale:** The schema column exists per the spec requirement; Phase 3 multi-household or per-household scoping can populate it meaningfully. For Phase 2, it functions as a category tag rather than a real FK.

### Decision F: Dedup window of 2 hours, log purge on each scheduler run

**Chosen approach:** On each `evaluateNudgePushRule` run, before pushing, filter the `push_notification_log` for entries newer than `now - 2 hours` for `(subscriptionId, entityType, entityId)`. Entries older than the window are not deleted immediately — they accumulate and the query ignores them via the `sentAt > cutoff` filter. Batch purge old log rows on each run (delete where `sentAt < now - 48 hours`) to prevent unbounded growth.

**Rationale:** Per D-045. A 48-hour purge threshold is conservative — any entry outside the 2-hour dedup window is irrelevant to push decisions but keeping them for 48 hours allows debugging. Simple row-level purge avoids the complexity of a separate cleanup job.

- The `web-push` npm package is already installed (it is — confirmed in `push.ts`).

## Phase 5: Nudge Push Scheduler

**Goal:** Add `evaluateNudgePushRule` to `jobs.ts`. On each cycle, compute active nudges via `repository.getNudgePayloads(now)`, check dedup, send push to all subscribed devices, handle 410 cleanup.

### 5.1 New rule function

```ts
export async function evaluateNudgePushRule(
  repository: InboxRepository,
  push: PushProvider,
  config: AppConfig,
  logger: FastifyBaseLogger,
  now: Date = new Date()
): Promise<void> {
  if (!push.isConfigured()) {
    logger.debug('nudge push rule: VAPID keys not configured, skipping');
    return;
  }

  const DEDUP_WINDOW_MS = 2 * 60 * 60 * 1000; // 2 hours
  const PURGE_RETENTION_MS = 48 * 60 * 60 * 1000; // 48 hours

  // Purge stale log entries (background hygiene)
  repository.purgeStalePushNotificationLog(PURGE_RETENTION_MS, now);

  const subscriptions = repository.listPushSubscriptions();
  if (subscriptions.length === 0) {
    logger.debug('nudge push rule: no push subscriptions');
    return;
  }

  const nudges = sortNudgesByPriority(repository.getNudgePayloads(now));
  if (nudges.length === 0) {
    logger.debug('nudge push rule: no active nudges');
    return;
  }

  for (const subscription of subscriptions) {
    for (const nudge of nudges) {
      const alreadySent = repository.hasPushNotificationLog(
        subscription.id, nudge.entityType, nudge.entityId, DEDUP_WINDOW_MS, now
      );
      if (alreadySent) continue;

      const notification = buildNudgeNotification(nudge, config.pwaOrigin);
      try {
        await push.send(
          { endpoint: subscription.endpoint, keys: { p256dh: subscription.p256dhKey, auth: subscription.authKey } },
          notification
        );
        repository.recordPushNotificationLog(subscription.id, nudge.entityType, nudge.entityId, now);
        logger.info({ subscriptionId: subscription.id, entityType: nudge.entityType, entityId: nudge.entityId }, 'nudge push delivered');
      } catch (error) {
        const statusCode = (error as { statusCode?: number }).statusCode;
        if (statusCode === 410) {
          repository.deletePushSubscription(subscription.endpoint);
          logger.info({ subscriptionId: subscription.id }, 'nudge push: 410 Gone — subscription removed');
          break; // subscription is gone; skip remaining nudges for it
        }
        logger.warn({ subscriptionId: subscription.id, entityId: nudge.entityId, error }, 'nudge push delivery failed (non-fatal)');
      }
    }
  }
}
```

function buildNudgeNotification(nudge: Nudge, pwaOrigin: string): NotificationPayload {

    body,
    url: `${pwaOrigin}/`,
    tag: `nudge-${nudge.entityType}-${nudge.entityId}`,

```ts
/// <reference lib="webworker" />
import { precacheAndRoute } from 'workbox-precaching';

declare const self: ServiceWorkerGlobalScope;

// Workbox injects precache manifest here at build time
precacheAndRoute(self.__WB_MANIFEST);

// ─── Push event handler ──────────────────────────────────────────────────────

self.addEventListener('push', (event: PushEvent) => {
  if (!event.data) return;

  let payload: { title?: string; body?: string; tag?: string; url?: string };
  try {
    payload = event.data.json() as typeof payload;
  } catch {
    payload = { title: 'Olivia', body: event.data.text() };
  }

  const title = payload.title ?? 'Olivia';
  const options: NotificationOptions = {
    body: payload.body ?? '',
    tag: payload.tag ?? 'olivia-nudge',
    icon: '/favicon.svg',
    data: { url: payload.url ?? '/' },
  };

  event.waitUntil(self.registration.showNotification(title, options));
});

// ─── Notification click handler ──────────────────────────────────────────────

self.addEventListener('notificationclick', (event: NotificationEvent) => {
  event.notification.close();

  const url: string = (event.notification.data as { url?: string } | null)?.url ?? '/';

  event.waitUntil(
    self.clients
      .matchAll({ type: 'window', includeUncontrolled: true })
      .then((clients) => {
        // Focus an existing Olivia window if open
        for (const client of clients) {
          if (client.url.startsWith(self.location.origin) && 'focus' in client) {
            return (client as WindowClient).focus();
          }
        }
        // Otherwise open a new window to home screen
        return self.clients.openWindow(url);
      })
  );
});
```

## Phase 7: PWA Opt-In UI

**Goal:** Show an in-app opt-in prompt in the nudge tray when at least one active nudge exists and push permission has not yet been granted on this device.

### 7.1 usePushOptIn hook

Create `apps/pwa/src/lib/push-opt-in.ts`:

```ts
import { useState, useEffect, useCallback } from 'react';

const DISMISSED_KEY = 'olivia-push-opt-in-dismissed';

export type PushOptInState = 'prompt' | 'granted' | 'denied' | 'unsupported' | 'dismissed';

export function usePushOptIn() {
  const [state, setState] = useState<PushOptInState>('unsupported');

  useEffect(() => {
    if (!('Notification' in window) || !('serviceWorker' in navigator)) {
      setState('unsupported');
      return;
    }
    const perm = Notification.permission;
    if (perm === 'granted') { setState('granted'); return; }
    if (perm === 'denied') { setState('denied'); return; }
    if (localStorage.getItem(DISMISSED_KEY)) { setState('dismissed'); return; }
    setState('prompt');
  }, []);

  const requestPermission = useCallback(async (): Promise<void> => {
    if (!('serviceWorker' in navigator)) return;
    const perm = await Notification.requestPermission();
    if (perm !== 'granted') {
      setState(perm === 'denied' ? 'denied' : 'dismissed');
      return;
    }
    setState('granted');
    try {
      const vapidRes = await fetch('/api/push-subscriptions/vapid-public-key');
      const { vapidPublicKey } = await vapidRes.json() as { vapidPublicKey: string | null };
      if (!vapidPublicKey) return;

      const reg = await navigator.serviceWorker.ready;
      const subscription = await reg.pushManager.subscribe({
        userVisibleOnly: true,
        applicationServerKey: urlBase64ToUint8Array(vapidPublicKey),
      });
      const subJson = subscription.toJSON() as { endpoint: string; keys: { p256dh: string; auth: string } };
      await fetch('/api/push-subscriptions', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ endpoint: subJson.endpoint, keys: subJson.keys }),
      });
    } catch (err) {
      console.warn('push subscription failed', err);
    }
  }, []);

  const dismiss = useCallback(() => {
    localStorage.setItem(DISMISSED_KEY, '1');
    setState('dismissed');
  }, []);

  return { state, requestPermission, dismiss };
}

// Utility: convert VAPID public key from base64url to Uint8Array
function urlBase64ToUint8Array(base64String: string): Uint8Array {
  const padding = '='.repeat((4 - (base64String.length % 4)) % 4);
  const base64 = (base64String + padding).replace(/-/g, '+').replace(/_/g, '/');
  const rawData = atob(base64);
  return Uint8Array.from([...rawData].map((c) => c.charCodeAt(0)));
}
```

## Notes on Nullable userId Column

Per D-045, the `push_subscriptions.user_id` column is added now and left `NULL` for all Phase 2 subscriptions. No code reads or writes this column in Phase 2 — it is purely a schema forward-compatibility placeholder. When Phase 3 per-member targeting is implemented, the column will be populated and the scheduler will filter subscriptions by userId.

The repository's `savePushSubscription` method accepts an optional `userId` parameter (defaulting to `undefined` → stored as NULL). The Phase 2 caller always omits it.

## Notes on In-Process Scheduler Placement

Per D-045, the nudge push scheduler runs as an in-process `setInterval` inside the API server, not as an external cron or job queue. This is consistent with the existing `startBackgroundJobs` pattern in `jobs.ts`. The scheduler is started in `server.ts` (or wherever `startBackgroundJobs` is called) and the cleanup function is registered with the Fastify `onClose` hook.

If the API process restarts between scheduler ticks, no pushes are lost — the next tick will find the same active nudges and the dedup log will prevent re-pushing items already delivered within the window.
